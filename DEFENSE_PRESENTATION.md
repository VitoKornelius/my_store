# Презентація для захисту

## Слайд 1. Назва проєкту

**Mercur Marketplace**

Multi-vendor маркетплейс із трьома ролями:

- покупець;
- продавець;
- адміністратор.

Проєкт побудований на Medusa та Mercur.

---

## Слайд 2. Ідея проєкту

Мета проєкту — створити маркетплейс, де різні продавці можуть розміщувати свої товари, а покупці переглядати каталог і додавати товари до кошика.

Це не просто інтернет-магазин, а платформа з кількома продавцями.

---

## Слайд 3. Основні ролі

**Покупець**

- переглядає каталог;
- шукає товари;
- фільтрує за категоріями;
- додає товари до кошика.

**Продавець**

- має власний кабінет;
- працює зі своїми товарами;
- бачить замовлення.

**Адміністратор**

- керує маркетплейсом;
- контролює продавців;
- має доступ до загальних налаштувань.

---

## Слайд 4. Архітектура

Проєкт складається з кількох частин:

- **Storefront** — інтерфейс покупця;
- **Vendor panel** — кабінет продавця;
- **Admin panel** — адміністративна панель;
- **Backend API** — Medusa/Mercur API;
- **PostgreSQL** — база даних;
- **Redis** — інфраструктурний сервіс.

Адреси локально:

- Storefront: http://localhost:8000
- Admin: http://localhost:7000/dashboard
- Vendor: http://localhost:7001/seller
- API: http://localhost:9000

---

## Слайд 5. Backend

Backend побудований на **Medusa 2** і **Mercur 2**.

Medusa відповідає за e-commerce основу:

- товари;
- варіанти;
- ціни;
- категорії;
- склади;
- API.

Mercur додає marketplace-логіку:

- продавців;
- seller roles;
- vendor API;
- зв'язок товарів із продавцями.

---

## Слайд 6. Що було реалізовано

У проєкті реалізовано:

- UI для покупців;
- каталог товарів;
- пошук і фільтри;
- модальне вікно товару;
- вибір варіанта;
- локальний кошик;
- демо-акаунти продавців і адміністраторів;
- велике наповнення тестовими даними.

Поточна демо-база:

- 12 продавців;
- 112 товарів;
- 34 категорії.

---

## Слайд 7. Дані та seed-скрипти

Для наповнення бази створені seed-скрипти:

- `seed` — базові налаштування Medusa;
- `seed:marketplace` — товари перших продавців;
- `seed:catalog` — великий каталог;
- `seed:more-products` — додаткові товари для всіх продавців.

Скрипти перевіряють товари за `handle`, тому їх можна запускати повторно без дублювання.

---

## Слайд 8. Безпека

У проєкті використано:

- авторизацію Medusa;
- RBAC для ролей;
- окремі API для покупця, продавця й адміністратора;
- CORS для обмеження доступних клієнтів;
- publishable API key для Store API;
- `.env` файли для секретів.

Важливо: демо-паролі використовуються лише для локальної презентації.

---

## Слайд 9. Обмеження демо-версії

У поточній версії:

- checkout у storefront поки демонстраційний;
- payment provider ще не підключений;
- email provider ще не налаштований;
- production storage для файлів ще не підключений;
- демо-паролі треба замінити перед реальним запуском.

Це нормальні обмеження навчального проєкту.

---

## Слайд 10. Висновок

У результаті створено робочий прототип multi-vendor маркетплейсу.

Проєкт демонструє:

- розділення ролей;
- роботу з реальним backend API;
- управління продавцями й товарами;
- наповнений каталог;
- готову основу для подальшого розвитку.

Наступні кроки:

- підключення повного checkout;
- інтеграція оплати;
- production-налаштування;
- покращення кабінету покупця.

---

# Короткий текст для виступу

Я розробив multi-vendor маркетплейс на базі Medusa та Mercur. На відміну від звичайного інтернет-магазину, тут є кілька типів користувачів: покупці, продавці та адміністратори.

Покупець може переглядати каталог, шукати товари, фільтрувати їх за категоріями та додавати до кошика. Продавець має власний кабінет, а адміністратор керує всім майданчиком.

Backend побудований на Medusa, яка дає e-commerce основу, і Mercur, який додає marketplace-функціональність. У базі є продавці, товари, категорії, варіанти, ціни та складські залишки.

Для демонстрації я наповнив систему тестовими даними: 12 продавців і 112 товарів. Також створив seed-скрипти, щоб дані можна було повторно відновити.

Проєкт є робочим прототипом. Основні частини вже працюють, а для production потрібно додати повний checkout, оплату, email-сервіс і замінити демонстраційні паролі.
