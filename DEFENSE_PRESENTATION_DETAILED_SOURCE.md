# Розгорнута презентація для захисту

## Слайд 1. Mercur Marketplace

Multi-vendor маркетплейс для покупців, продавців і адміністраторів.

- курсова робота;
- стек: Medusa, Mercur, React, PostgreSQL;
- робочий прототип із наповненими демо-даними.

## Слайд 2. Актуальність теми

Маркетплейси є однією з основних моделей сучасної електронної комерції.

- один майданчик об'єднує багато продавців;
- покупець отримує ширший вибір товарів;
- продавці можуть працювати незалежно;
- адміністратор керує платформою централізовано;
- така архітектура ближча до реальних e-commerce систем.

## Слайд 3. Мета проєкту

Мета — створити робочий прототип multi-vendor маркетплейсу.

- реалізувати окремі інтерфейси для покупця, продавця й адміністратора;
- підключити frontend до реального backend API;
- налаштувати marketplace-логіку продавців і товарів;
- наповнити систему демонстраційними даними;
- показати базову архітектуру e-commerce платформи.

## Слайд 4. Ролі користувачів

У системі розділені сценарії для трьох типів користувачів.

Покупець:
- переглядає каталог;
- шукає товари;
- фільтрує за категоріями;
- додає товари до кошика.

Продавець:
- має власний кабінет;
- керує своїми товарами;
- працює із замовленнями;
- належить до seller-ролі.

Адміністратор:
- керує маркетплейсом;
- контролює продавців;
- бачить товари та налаштування;
- має доступ до admin panel.

## Слайд 5. Загальна архітектура

Проєкт поділено на окремі застосунки та backend API.

- Storefront звертається до Store API;
- Vendor Panel звертається до Vendor API;
- Admin Panel звертається до Admin API;
- Backend працює з PostgreSQL і Redis;
- усі частини запускаються в одному monorepo.

## Слайд 6. Backend

Backend побудований на Medusa 2 і Mercur 2.

Medusa:
- товари;
- варіанти;
- ціни;
- категорії;
- склади.

Mercur:
- продавці;
- vendor API;
- seller roles;
- зв'язок товарів із продавцями.

## Слайд 7. Storefront для покупців

Окремий React/Vite інтерфейс працює з реальним Store API.

- каталог отримує товари з backend;
- є пошук і фільтрація;
- картки показують назву, ціну, категорію і продавця;
- у модальному вікні можна вибрати варіант товару;
- кошик зберігається локально в браузері.

## Слайд 8. Демо-дані

Проєкт наповнений реалістичними прикладами для демонстрації.

- 12 продавців;
- 112 товарів;
- 34 категорії;
- товари мають ціни, варіанти й залишки;
- кожен демо-товар прив'язаний до продавця;
- дані можна повторно відновити через seed-скрипти.

## Слайд 9. Seed-скрипти

Для демонстрації створено скрипти автоматичного наповнення бази.

- seed — базові налаштування Medusa;
- seed:marketplace — товари перших продавців;
- seed:catalog — великий каталог;
- seed:more-products — додаткові товари для всіх продавців;
- перевірка за handle запобігає дублюванню товарів.

## Слайд 10. Безпека

Backend використовує базові механізми безпеки Medusa і Mercur.

- авторизація Medusa;
- RBAC для ролей;
- окремі Store, Admin і Vendor API;
- Storefront не має доступу до Admin API;
- Store API використовує publishable API key;
- CORS обмежує клієнти;
- секрети зберігаються в .env.

## Слайд 11. Обмеження і подальший розвиток

Поточна версія є робочим навчальним прототипом.

- checkout у storefront поки демонстраційний;
- payment provider ще не підключений;
- email provider не налаштований;
- production file storage ще не підключений;
- демо-паролі потрібно замінити;
- можна додати кабінет покупця та історію замовлень.

## Слайд 12. Висновок

Створено робочий прототип multi-vendor маркетплейсу.

- реалізовано три ролі користувачів;
- frontend підключений до реального backend API;
- система наповнена демонстраційними даними;
- є окремі панелі покупця, продавця й адміністратора;
- архітектура дозволяє розвивати проєкт до production-рівня.

Фінальна теза: проєкт демонструє не лише інтерфейс магазину, а основу marketplace-системи з продавцями, товарами, ролями та API.
